Key Decisions
Clinic feature will follow the exact same architectural pattern as Shops → Products → Orders.
Doctor role should be a new user role (e.g. DOCTOR) distinct from SHOP_OWNER.
Clinic timeslots will be a new model (Timeslot) tied to a Clinic, each with start/end time and is_booked boolean.
Booking will replace the existing Order flow but with date‑time selection instead of cart/checkout.
A new dedicated clinics Django app will mirror the shops app structure.
Next Steps
Create the clinics Django app with models: Clinic, Timeslot, Booking, ClinicRating, ClinicNotification.
Add DOCTOR role to the User model (or use a separate profile).
Implement backend APIs: CRUD clinics, timeslot management, booking creation, revenue aggregation.
Build frontend pages: doctor dashboard, clinic setup form, clinic public detail page, user booking flow, revenue dashboard.
Wire notifications into the booking lifecycle (new booking → doctor, status change → user).
Critical Context
Backend is Django DRF with JWT auth (the existing auth system). A new clinics app will reuse the same patterns as shops and orders.
Frontend is vanilla HTML/CSS/JS — no JS framework. All pages are static HTML served by nginx, communicating via the shared api.js client.
The platform is in active development – careful to keep existing apps unchanged (sessions 8–13 for shops/orders are already in production).
